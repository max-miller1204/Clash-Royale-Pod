# KataCR class-name mapping → `CardPlay`

**Status:** approved (design); implementation pending.
**Owner:** max
**Date:** 2026-04-30
**Related:** `docs/TODO.md` § "Data quality" — name-mapping layer for KataCR class names.

## Problem

The trained YOLO model (`output/models/crpod_v1_best.pt`, ~150 KataCR classes) emits class strings in KataCR's convention — hyphenated, often singular (`the-log`, `spear-goblin`). The downstream lookup `card_cost()` in `src/crpod/constants.py` expects the project's underscore convention (`log`, `spear_goblins`). Two consequences:

1. **Every card play silently gets `default=3` elixir.** `card_cost("the-log")` misses CARD_COSTS, falls back to 3 instead of 2. Elixir trade and EV signals are quietly biased.
2. **Non-card detections become phantom card plays.** `_parquet_to_replay` (src/crpod/dataset/huggingface.py:44) constructs a `CardPlay` from *every* `Detection`, including towers, health bars, projectiles, and UI elements. Those become bogus 3-cost plays in the replay stream.

Both bugs corrupt the data the EV model trains on. They were masked until now because YOLO weights didn't exist; now that they do, this is on the critical path.

## Goals

- `card_cost()` lookups hit for every card the detector emits.
- Non-card detections are dropped before becoming `CardPlay` events.
- New, unmapped KataCR classes are surfaced (warned once), not silently swallowed.
- Tests fail loudly if the trained model gains/loses classes without the mapping being updated.

## Non-goals

- Filling in costs for the 4 unconfirmed champions (`boss_bandit`, `rune_giant`, `spirit_empress`, `terry`). They keep `default=3` until Supercell publishes costs.
- Replacing `card_cost()` itself or restructuring `CARD_COSTS`. Existing keys/values stay.
- Filtering at the detector layer — `YoloDetector` stays a thin pass-through.

## Design

### Module layout

New module **`src/crpod/detection/cards.py`** owns the KataCR-class → CardPlay boundary:

```python
from crpod.constants import card_cost
from crpod.detection.yolo import Detection
from crpod.types import CardPlay

KATACR_TO_CARD: dict[str, str] = {
    "the-log": "log",
    "spear-goblin": "spear_goblins",
    # ... populated against katacr_classes.txt at implementation time
}

KATACR_NON_CARD: frozenset[str] = frozenset({
    "king-tower",
    "queen-tower",
    "king-tower-bar",
    # ... towers, HP bars, projectiles, UI, emote
})

def to_card_play(det: Detection) -> CardPlay | None:
    """Convert a raw detection into a CardPlay, or None if it's not a card play."""
    if det.cls in KATACR_TO_CARD:
        canonical = KATACR_TO_CARD[det.cls]
        cx, cy = det.center
        return CardPlay(
            frame=det.frame,
            card=canonical,
            x=int(cx),
            y=int(cy),
            side=_infer_side(int(cy)),
            elixir_cost=card_cost(canonical),
        )
    if det.cls in KATACR_NON_CARD:
        return None
    _warn_unknown(det.cls)  # warn-once per class
    return None
```

The two parallel structures keep separation of concerns clean: name aliasing lives here, cost data stays in `CARD_COSTS`. Adding a new card means one row in `KATACR_TO_CARD` (if its KataCR name differs from the canonical key) and one row in `CARD_COSTS`. Adding a new non-card class means one row in `KATACR_NON_CARD`.

### Source of truth: `src/crpod/detection/katacr_classes.txt`

A checked-in snapshot of the trained model's `model.names` — one class per line, in class-id order. This is the authoritative list of strings the deployed weights can emit. Tests assert that the mapping (`KATACR_TO_CARD ∪ KATACR_NON_CARD`) covers it exactly.

Generated by a one-shot **`scripts/dump_katacr_classes.py`** that loads `output/models/crpod_v1_best.pt` and writes the file. Run once on a machine that has the .pt; commit the result.

### Caller change

`src/crpod/dataset/huggingface.py` already has a private helper `_detection_to_card_play(det)` (huggingface.py:36) that builds a CardPlay from a Detection. The change: move this helper into the new `crpod.detection.cards` module, rename to `to_card_play`, and (a) consult `KATACR_TO_CARD` for the canonical card name, (b) return `None` for non-card classes. The `_infer_side` helper relocates with it. The caller (whatever currently consumes `_detection_to_card_play`) filters `None`s.

### Unknown-class handling

`_warn_unknown` uses a module-level `set` of already-warned class names so each unknown only logs once per process. Logged via `logging.getLogger(__name__).warning(...)`. Rationale: an unknown class almost always means a card we forgot to map (or a model retrain with new classes); we want it visible without spamming logs at 30 fps.

The test suite's coverage check (below) catches this earlier — at PR time — but the runtime warning is a safety net for cases where weights and code drift apart in deployment.

## Testing

New file **`tests/test_cards.py`** with three test classes:

1. **Coverage** — `set(katacr_classes.txt) == set(KATACR_TO_CARD.keys()) | KATACR_NON_CARD`. Fails if the snapshot adds/removes a class without the mapping being updated. Skipped with a clear message if `katacr_classes.txt` doesn't exist yet.
2. **Validity** — every value in `KATACR_TO_CARD` is either (a) a key in `CARD_COSTS`, or (b) one of the 4 known-unconfirmed champions. Catches typos in alias targets.
3. **Behavioral** — `to_card_play()` on a synthetic card detection returns a `CardPlay` with the right canonical name and cost; on a synthetic tower detection returns `None`; on a synthetic unknown class returns `None` and emits a warning (asserted via `caplog`).

No new fixtures or weights needed — tests construct `Detection` instances directly.

## Implementation notes

The mapping rows in `KATACR_TO_CARD` and `KATACR_NON_CARD` get populated during implementation, not design. Procedure:

1. Run `scripts/dump_katacr_classes.py` to produce `katacr_classes.txt`.
2. For each line, decide: card → add alias row pointing at the matching `CARD_COSTS` key (or note if a `CARD_COSTS` key is missing); non-card → add to `KATACR_NON_CARD`.
3. Run the coverage test; fix gaps; iterate.

Most of the work is mechanical. Estimated 1-2 hours including running the dump on a remote machine.

## Risks

- **The .pt isn't on the local dev machine.** The dump script needs to be run on the brev instance (or wherever the .pt lives) and the resulting `katacr_classes.txt` brought back. Until then, the coverage test stays skipped. Mitigation: add a CI job later that fails if the file is missing, so the gap doesn't become permanent.
- **KataCR class names use hyphens; YAML/Python keys generally don't.** The dict-key-with-hyphen pattern is unusual but legal Python; just means we can't use attribute access. Acceptable.
- **Future model retrains will change class IDs.** The snapshot is keyed by name (not ID), so re-ordering is fine; only adds/removes need a mapping update. Coverage test catches this.
