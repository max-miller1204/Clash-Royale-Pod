How blunder detection works

The wave-3 deliverable: a single statistical rule that turns the EV model's predictions into a list of "you should have played something else" callouts.

The rule, in one line

A play is a blunder when (mediancard − EVpred) / σcard > 1

In plain English: the model thinks the outcome you're about to get from this card is more than one standard deviation worse than what the card typically returns.

Three pieces, each from a different stage

VariableWhere it comes fromWhat it represents
EVpred EvModel.predict(interaction_features) LightGBM's predicted "(friendly princess Δhp) − (enemy princess Δhp)" for the 40-frame window starting at this play.
mediancard EvModel.per_card_stats[card][0] The training-fold median of that quantity, grouped by the card you just played (anchored on the friendly side's first card per interaction).
σcard EvModel.per_card_stats[card][1] The training-fold population standard deviation for the same card.

per_card_stats is computed once at training time and persisted inside the .joblib artifact. Inference never needs the training data on disk — just the model and the stats dict.

Why a card-aware threshold

A naive blunder rule would just compare predicted EV to global median: "any play scoring below the typical play is bad." But that punishes high-variance cards unfairly.

Example. Mega Knight is a defensive card — when you play it the typical outcome is "you lost a little tower HP because you were already getting tower-dived." Its training-fold median is around -300, with σ ≈ 400.
Knight is a cheap tempo card — typical outcome ≈ 0 (it doesn't usually win the trade outright but doesn't lose much either). Its σ is ≈ 100.

An EV prediction of -200 is:
  • Mega Knight: (−300 − (−200)) / 400 = −0.25σ → not a blunder (better than typical)
  • Knight: (0 − (−200)) / 100 = +2.0σ → BLUNDER (much worse than typical)
Same predicted outcome, completely different judgment — because the rule asks "is this bad for this card?"

Why > 1σ specifically

One standard deviation captures roughly the worst 16 % of plays for a normally-distributed card (the long left tail). In practice:

  • Most matches surface 0–3 blunders — pod-shippable density.
  • Cranking the threshold to 2σ would surface only the worst 2 % — great for "did I make any obvious crime?" but too sparse for coaching feedback.
  • Dropping to 0.5σ would flag every below-average play as a blunder — which is correct in a literal sense but useless as advice.

The threshold is hardcoded as _SIGMA_THRESHOLD = 1.0 in src/crpod/analysis/blunders.py. Bumping it is a one-line change if a different sensitivity is wanted later.

What gets filtered out

The detector intentionally skips:

  • Cards with fewer than 5 training-fold samples. These are already filtered out of per_card_stats by compute_per_card_stats in crpod.modeling.ev — a card the detector has never seen 5 times can't be judged.
  • Cards the model has literally never seen. Missing-key lookups return None from per_card_stats.get(card); the detector silently skips the play rather than fabricating a comparison.
  • Cards whose training distribution has zero std. Mathematically pathological — would divide by zero. In practice this almost never happens since statistics.pstdev only returns zero for an exactly-constant series.

What the output looks like

[
  {
    "play_idx": 17,
    "card": "hog_rider",
    "ev_predicted": -250.3,
    "per_card_median": 80.0,
    "sigma_below": 2.65
  },
  {
    "play_idx": 4,
    "card": "knight",
    "ev_predicted": -45.0,
    "per_card_median": 12.0,
    "sigma_below": 1.42
  }
]

Plays are ordered worst-first. play_idx is the index into result.interactions so you can recover the full window context (start frame, enemy plays, elixir trade) downstream.

What it doesn't (yet) do

  • Doesn't tell you what you should have played instead. Counterfactual reasoning would need a per-game-state action-value table, which is a whole separate research project. Wave 3's blunder list is descriptive ("this was bad"), not prescriptive ("you should have done X").
  • Doesn't account for game state / score. Down two towers in overtime, conservative plays look worse than they "should" — but the EV target itself doesn't condition on score, so the blunder rule doesn't either.
  • Treats the friendly anchor card as the only one that matters per interaction. If you played Knight + Musketeer and the model thinks the outcome was awful, only Knight gets blamed (it's the anchor). The CLI surfaces the interaction context so you can manually attribute, but the rule itself is single-card.

Source: src/crpod/analysis/blunders.py. Tests: tests/test_blunders.py (8 cases — zero, multi, threshold-strict, <5-samples, unknown-card, zero-σ, length-mismatch, play-idx alignment).
