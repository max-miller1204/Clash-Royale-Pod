The post-game report

A single HTML file the pod can email, AirDrop, drop in Slack, open offline — no network, no dependencies, no build step.

The file

output/analysis/<replay_id>/report.html

Generated by crpod.visualization.report.render_report. Bundles the summary, top blunders, and both PNG visualizations into a single self-contained file. Open it in Chrome, Safari, Firefox — works identically. No JavaScript, no external CSS, no network requests. Drag into a browser tab and you're done.

Why self-contained?

  • Sharing. AirDrop / Slack-drop the single file to a teammate and they see exactly what you see, no zip, no folder.
  • Permanence. Embedded base64 PNGs survive rename / move / archival — the report stays viewable years later even if the sibling placements.png is gone.
  • No CDN, no JS frameworks. No supply-chain risk, no "did this work back in 2024?" worry. Plain HTML + inline CSS.
  • Cost. A 30-replay analysis produces 30 report.html files at roughly 80–150 KB each (the PNGs are the size driver). Trivial.

Graceful degradation

If placements.png or tempo.png failed to render (the CLI logs [warn] viz skipped: ... when that happens), the report still ships — the missing image slot becomes an in-line placeholder div with the explanation. The report does not crash on missing siblings.

Same with blunders: if no EV model was supplied (so result.blunders is empty), the table shows a "No blunders flagged" placeholder rather than rendering an empty <table>.

What it deliberately doesn't have

  • No interactive elements. No sortable columns, no timeline scrubbing, no JS filters. A "post-game report" is read once after the match — interactivity would be over-engineering.
  • No replay-link-out. The report doesn't link back to in-game replays. The HF dataset doesn't ship deeplinks and the pod's own videos live on local disks.
  • No comparison-across-matches. Each report is one-replay. Pod-level aggregation could live in a separate dashboard later — but the per-match report is the unit.
